ðŸ”„ Implementation Phases

Phase 0: Baseline Hardening
  • Extract per-topic log to replication/log.py, add commit index
  • Implement idempotent append with ID cache
  • Add message ID in publisher and dedup on broker
Acceptance: Single node passes publish/subscribe/history; persisted offsets strictly increase.
Phase 1: RPC + Cluster Config
  • Implement lightweight RPC server/client over TCP for Raft messages
  • Create static config/cluster.yaml
  • Node boots with role Follower, term/state persisted to disk
Acceptance: Nodes exchange heartbeats (no election yet).
Phase 2: Raft Election
  • Implement RequestVote with randomized timeouts
  • Leader election mechanism
  • Leader sends heartbeats to maintain authority
Acceptance: With 3 nodes, one leader elected; on leader kill, new leader elected.
Phase 3: Log Replication + Commit
  • AppendEntries carries topic message entries
  • Majority quorum commit; followers persist then ack
  • Leader serves PUB, others REDIRECT to leader host:port
Acceptance: Publish to any node; messages replicated to majority and survive leader crash.
Phase 4: Client Delivery Semantics
  • Deliver only committed entries to subscribers
  • Support HISTORY from committed log
  • Ensure no message loss/duplication across failover
Acceptance: No message loss/duplication across leader failover; offsets strictly monotonic.
Phase 5: Time Synchronization
  • SNTP offset estimation between nodes
  • Maintain per-peer clock_offset
  • Add Lamport clocks to entries
  • Bounded reorder on delivery using (lamport, ts)
Acceptance: Out-of-order arrivals reorder correctly; log skew metrics exposed.
Phase 6: Recovery and Catch-up
  • On rejoin, follower requests missing entries from leader
  • Match index and backfill mechanism
  • Handle network partition recovery
Acceptance: Stop a follower, publish N messages, restart follower; it catches up.
Phase 7: Observability and Tests
  • Structured logs with appropriate log levels
  • Simple metrics counters for monitoring
  • Integration tests for failover scenarios
  • Partition tests (simulate dropped RPCs)
Acceptance: Full test suite passes; metrics accurately reflect system state.
